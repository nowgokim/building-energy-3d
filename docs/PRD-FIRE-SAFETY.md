# PRD: 3D 건물 소방 방재 분석 플랫폼 (Fire Safety Extension)

**문서 버전**: 1.1
**작성일**: 2026-03-24
**최종 업데이트**: 2026-03-25
**상태**: Phase F0 완료 / Phase F1 완료 / Phase F2~F4 계획
**기반 플랫폼**: PRD v1.1 (3D 건물 에너지 시뮬레이션 지도 플랫폼)

---

## 1. 개요 & 배경

### 1.1 문제 정의

한국의 도시 소방 방재 체계는 다음과 같은 구조적 한계를 가진다.

- **데이터 분절**: 건축물대장, GIS 공간정보, 소방시설 점검 결과, 기상 데이터가 각각 별도 시스템에 산재되어 있어 통합 위험 평가가 불가능하다.
- **정적 위험 평가**: 현행 소방 안전 관리는 개별 건물 단위의 정기 점검에 의존한다. 도시 스케일에서 어느 구역이 화재 위험이 집중되어 있는지 시각적으로 파악할 도구가 없다.
- **확산 경로 예측 부재**: 밀집 노후 건물 지역에서 화재 발생 시 인접 건물로의 확산 경로를 사전에 시뮬레이션하는 공공 도구가 존재하지 않는다.
- **대피 경로 최적화 미흡**: 소방관과 주민의 대피 경로가 현장 상황에 따라 실시간으로 달라지나, 이를 3D 공간 위에서 즉시 계산해 제공하는 의사결정 지원 도구가 없다.
- **보험 인수 비효율**: 화재보험사 언더라이터가 개별 건물의 위험도를 산정할 때 공간적 맥락(인접 건물 밀도, 소방서 접근 시간 등)을 반영하지 못한다.

### 1.2 시장 맥락

소방 방재 데이터 분석 시장은 에너지 분야와 다른 구조적 특성을 가진다. B2G(소방청, 지자체) 발주 사업이 주를 이루며, B2B 화재보험 분야는 글로벌 InsurTech 트렌드에 따라 데이터 기반 위험도 산정 수요가 빠르게 성장하고 있다.

국내 화재 현황을 보면, 2023년 기준 전국 화재 건수는 약 38,000건으로 이 중 건물 화재가 50% 이상을 차지한다(소방청 국가화재정보시스템, 2023). 특히 1990년대 이전에 지어진 조적조·목구조 건물과 불법 증축이 많은 고밀 주거지에서의 화재는 확산 속도가 빠르고 피해 규모가 크다.

본 플랫폼이 이미 수집한 서울 전역 766,386건의 건물 데이터와 PostGIS 공간 DB는 소방 방재 분석의 핵심 인프라로 즉시 전용 가능하다. 에너지 플랫폼이 구축한 구조 유형(structure_type), 건축년도(built_year), 용도(usage_type), 층수(floors_above), 높이(height)는 화재 위험도 산정의 1차 입력 변수와 정확히 일치한다.

### 1.3 솔루션 개요

기존 3D 건물 에너지 플랫폼 위에 소방 방재 분석 레이어를 추가하는 확장(extension) 방식으로 개발한다. 신규 데이터 파이프라인 구축 비용을 최소화하면서 소방관, 지자체 소방담당 공무원, 건물 소방안전관리자, 화재보험사 언더라이터가 각각의 업무에서 즉시 활용할 수 있는 의사결정 지원 도구를 제공한다.

## 2. 목표 & 성공 지표

### 2.1 프로젝트 목표

1. 서울 전역 766K 건물에 대한 화재 위험도 점수를 자동 산정하고 3D 지도에 시각화한다.
2. 도시 밀도와 건물 군집을 분석하여 화재 취약 지구를 식별하고 우선순위화한다.
3. BFS(너비우선탐색) 기반 화재 확산 시뮬레이션으로 인접 건물 피해 경로를 예측한다.
4. pgRouting과 OSM 도로망을 활용하여 소방차 접근 경로 및 주민 대피 경로를 실시간 계산한다.
5. 소방관, 공무원, 건물 관리자, 보험사가 각자의 워크플로우에서 바로 사용할 수 있는 UI를 제공한다.

### 2.2 성공 지표 (KR)

| 지표 | 목표 | 측정 방법 |
|------|------|----------|

| 위험도 지도 렌더링 | 5초 이내 (도시 스케일) | Lighthouse 측정 |

## 3. 타겟 사용자 & 페르소나

### 3.1 소방관 현장 지휘관 (Primary)

**"최 팀장" - 서울 마포소방서 현장대응팀 팀장, 15년 경력**

화재 신고 접수 후 출동 전 또는 현장 도착 직전에 해당 건물과 인근 지역의 구조적 위험 정보를 빠르게 파악해야 한다.

- **Pain**: 현재 출동 지령서에는 주소와 신고 내용만 있다. 건물이 몇 년도에 지어진 조적조인지, 인접 건물과의 거리가 얼마나 되는지, 주변에 위험물 저장 시설이 있는지를 현장에서 즉각 확인할 방법이 없다.
- **Need**: 출동 주소를 입력하면 해당 건물의 구조 유형·층수·건축년도와 인접 건물의 화재 위험도를 3D 지도 위에서 10초 안에 보여주는 도구. 소방차 접근 가능한 경로와 주민 대피 경로까지 함께 제공되면 이상적이다.
- **Value**: 현장 초기 대응 전략 수립 시간 단축, 인명 피해 최소화.
- **사용 환경**: 소방서 대기실 대형 모니터 또는 출동 차량 태블릿.

### 3.2 지자체 소방담당 공무원 (Primary)

**"김 주무관" - 서울시 소방재난본부 예방과 6년차**

관할 구역 내 노후·고위험 건물에 대한 연간 소방 점검 계획을 수립하고, 소방청에 예산 및 정책 보고서를 제출한다.

- **Pain**: 관할 내 어떤 건물이 가장 위험한지 순위를 매기는 작업을 매번 엑셀로 수작업한다. 지도 위에서 위험 건물의 공간적 분포를 보여주는 자료가 없어 보고서 작성에 며칠이 걸린다.
- **Need**: 구·동별 화재 위험 현황을 한눈에 보는 대시보드. 위험도 상위 건물 목록과 CSV 내보내기. 소방 점검 우선순위 지정에 쓸 수 있는 지도 시각화 자료.
- **Value**: 데이터 기반 점검 계획 수립, 보고서 작성 시간 단축, 예산 투입 근거 확보.

### 3.3 건물 소방안전관리자 (Secondary)

**"박 과장" - 강남구 복합 상업 건물 소방안전관리자 2급 자격, 8년 경력**

대형 건물의 소방 시설 유지보수와 연간 자체 점검을 담당하며, 건물 소유주에게 소방 안전 투자 우선순위를 제안한다.

- **Pain**: 우리 건물의 화재 위험도가 주변 건물과 비교해서 어느 수준인지 객관적인 근거가 없다. 건물주에게 리트로핏(스프링클러 추가 설치, 방화구획 강화 등)을 제안할 때 정량적 데이터가 부족하다.
- **Need**: 내 건물의 위험도 점수 상세 내역과 인근 동일 용도 건물과의 비교. 소방 시설 개선 시 위험도 점수 변화 시뮬레이터.
- **Value**: 건물주 설득력 강화, 소방 투자 ROI 산정 근거.

### 3.4 화재보험사 언더라이터 (Secondary)

**"이 팀장" - 대형 손해보험사 기업 화재 언더라이팅팀 7년차**

기업 건물의 화재보험 가입 심사 시 보험료를 산정하고 인수 여부를 결정한다.

- **Pain**: 현재 위험도 평가는 건물 자체 속성(구조, 연면적, 용도)에만 의존한다. 인접 건물 밀도, 소방서 접근 시간, 과거 화재 발생 이력 같은 공간적 맥락이 반영되지 않아 손해율 예측 오차가 크다.
- **Need**: 계약 건물 주소를 입력하면 위험도 점수와 그 산출 근거(구조 유형, 건축년도, 인접 건물 밀도, 소방서 거리)를 API로 받아 자체 언더라이팅 시스템에 연동하는 인터페이스.
- **Value**: 언더라이팅 정확도 향상, 역선택 리스크 감소, 적정 보험료 산정.

## 4. 기능 요구사항 (Phase별)

### Phase F0 — 화재 위험도 기본 점수 ✅ 완료

에너지 플랫폼의 기존 건물 속성만으로 위험도를 산정하는 1차 모델. 신규 데이터 수집 없이 즉시 구현 가능하다.

| ID | 기능 | 설명 | 우선순위 | 상태 |
|----|------|------|---------|------|
| FF0-01 | 화재 위험도 점수 산정 | structure_type(40점) + built_year(30점) + usage_type(20점) + height(10점) = 100점 | P0 | ✅ |
| FF0-02 | 위험도 등급 분류 | HIGH≥60 / MEDIUM 35~59 / LOW<35 | P0 | ✅ |
| FF0-03 | building_fire_risk Materialized View | PostGIS에 위험도 점수·등급 사전 계산 저장 | P0 | ✅ |
| FF0-04 | 화재 위험도 API | GET /api/v1/fire/risk, /fire/risk/{pnu}, /fire/stats | P0 | ✅ |
| FF0-05 | 히트맵 시각화 | 3D 지도에 위험도 등급별 색상 오버레이 (VWorld 뷰어) | P0 | ✅ |

**F0 점수 산정 로직 요약**

| 변수 | 배점 | 높은 위험 조건 |
|------|------|--------------|
| structure_type | 40점 | 목구조·조적조(40) > 경량철골(30) > RC·철골(10) |
| built_year | 30점 | 1970년 이전(30) > 1980년대(25) > 1990년대(20) > 2000년대(10) > 2010년 이후(5) |
| usage_type | 20점 | 위험물취급·창고(20) > 노유자·의료(18) > 판매·업무(15) > 주거(10) > 교육(8) |
| height (floors_above) | 10점 | 15층 초과(10) > 5~15층(7) > 5층 이하(5) |

## 5. 비기능 요구사항

### 5.1 성능

| 항목 | 목표 | 비고 |
|------|------|------|
| 화재 위험도 지도 초기 로딩 | 5초 이내 | 도시 스케일, Tile 스트리밍 |
| 건물 클릭 → 위험도 상세 | 2초 이내 | Materialized View KNN 조회 |
| DBSCAN 군집 분석 (서울 전역) | 30초 이내 | 배치 처리, Celery 비동기 |
| BFS 확산 시뮬레이션 (1km² 구역) | 10초 이내 | API 요청 후 Celery 처리 |
| pgRouting 경로 계산 | 3초 이내 | 사전 그래프 캐싱 |
| 기상 데이터 갱신 | 10분 주기 | Redis TTL |
| 동시 사용자 | 100명 (MVP) → 500명 (B2G 계약 시) | FastAPI + Gunicorn |

### 5.2 가용성 & 신뢰성

B2G 계약 환경에서는 소방 대응 중 시스템 다운이 허용되지 않는다.

- 서비스 가용성 목표: 99.5% 이상 (월 기준 3.6시간 이내 다운)
- Materialized View는 데이터 변경과 무관하게 항상 조회 가능
- API 서버 장애 시 최근 캐시 결과 fallback 제공
- 화재 확산 시뮬레이션 및 경로 계산은 Celery 큐를 통한 비동기 처리로 API 서버 부하 분리

### 5.3 보안

소방 위험도 정보는 악용 가능성이 있으므로 접근 제어가 필요하다.

- 화재 위험도 원시 데이터 API: API 키 인증 필수 (공개 히트맵 뷰와 구분)
- 화재 확산 시뮬레이션 API: 인증된 사용자(소방관, 공무원, 계약 기관)만 접근
- 소방서 위치·담당 구역 데이터: 공개 가능 (이미 공공데이터포털에서 제공)
- 실시간 화재 신고 데이터: 수신 후 즉시 암호화 저장, 30일 후 자동 삭제
- HTTPS 전용, CORS 화이트리스트 제한

### 5.4 데이터 정확성

위험도 점수는 의사결정의 근거가 되므로 불확실성을 명시해야 한다.

- 모든 위험도 점수에 신뢰도 등급(A: 건축물대장 확인 / B: 추정 / C: 기본값) 부여
- 건축물대장 미매칭 건물에는 "데이터 불완전" 표시
- 소방 시뮬레이션 결과 화면에 "이 결과는 근사 모델 기반 예측입니다. 실제 상황은 다를 수 있습니다" 고지 문구 표시 의무화

4. **건축물대장 구조 정보 신뢰도**: 1990년대 이전 건물의 structure_type 필드 누락률이 15~25%로 높다. 누락 시 RC조(중위 위험) 기본값 적용 후 신뢰도 등급 C 부여.

## 7. 비목표 (Scope 제외)

다음 항목은 이 문서의 범위에 포함되지 않는다. 구현 시 별도 RFC가 필요하다.

- **실내 화재 시뮬레이션**: 건물 내부 공간의 CFD(전산유체역학) 연기·열 확산 시뮬레이션. 건물 간 외부 확산에만 집중한다.
- **소방 장비 재고 관리 시스템**: 소방서 내부 운영 시스템과의 연동. 데이터 조회만 허용하며 쓰기 연동은 별도 계약 사항이다.
- **건물 내부 BIM 연동**: IFC/Revit 파일 기반 내부 구조 분석. 외피 기반 위험도 산정에만 집중한다.
- **자연재해 시뮬레이션**: 지진, 홍수, 산사태 등 다른 재난 유형. 화재 방재만 다룬다.
- **모바일 네이티브 앱**: 현장 소방관용 iOS/Android 앱. 웹 반응형(태블릿 최적화)으로 대체한다.
- **법적 효력 있는 소방 점검 문서 생성**: 소방시설법에 따른 공식 점검 보고서 양식. 참고 자료 생성만 지원한다.
- **전국 확장**: Phase F1~F4는 서울에 집중한다. 전국 확장은 별도 Phase에서 논의한다.

## 8. 비즈니스 모델

### 8.1 B2G — 소방청·지자체

공공기관 대상 연간 구독 또는 용역 계약 방식이다.

**타겟 고객**: 서울시 소방재난본부, 각 자치구 소방안전과, 소방청 정보통신담당관

**가치 제안**: 기존 GIS 소방 정보 시스템을 대체하거나 보완하는 분석 레이어 제공. 공무원의 점검 계획 수립 시간 단축 및 데이터 기반 예산 편성 근거 확보.

**가격 모델**:
- 구(區) 단위 라이선스: 약 2,000~5,000만원/년 (25개 자치구 기준)
- 서울시 전체 계약: 별도 협의

**진입 전략**: Phase F1 완료 후 1~2개 자치구 파일럿 사업(6개월) → 성과 데이터 기반 전체 계약 확장.

### 8.2 B2B — 화재보험사

API 호출 기반 종량제 또는 계정 구독 방식이다.

**타겟 고객**: 삼성화재·현대해상·KB손해보험 등 기업 화재보험 언더라이팅 부서

**가치 제안**: 언더라이터가 기존 시스템에서 주소 하나를 입력하면 위험도 점수, 인접 건물 밀도, 소방서 접근 시간을 API로 즉시 수신. 역선택 리스크 감소 및 손해율 예측 정확도 향상.

**가격 모델**:
- API 호출당 과금: 건당 50~100원 (연간 수십만 건 조회 가정)
- 연간 구독: 2,000만원/년 이상 (무제한 조회)

**진입 전략**: Phase F1 완료 후 1개 보험사와 파일럿 API 연동 → 6개월 손해율 데이터 검증 → 상용 계약.

### 8.3 SaaS — 건물 소방안전관리자

웹 기반 셀프서비스 구독 모델이다.

**타겟 고객**: 소방안전관리자 선임 의무 대상 건물 (연면적 5,000m² 이상 등) 약 10만 개소

**가치 제안**: 자기 건물의 위험도 점수 상세 내역, 인근 건물 비교, 리트로핏 시뮬레이터에 접근. 건물주 보고 자료 자동 생성.

**가격 모델**:
- 기본 플랜: 건물당 월 3~5만원 (위험도 조회 + 리포트)
- 프리미엄: 건물당 월 10만원 (화재 확산 시뮬레이션 + 대피 경로 포함)

**진입 전략**: Phase F2 완료 후 공개 베타 → 소방안전관리자 커뮤니티 채널 직접 마케팅.

## 9. 리스크 & 제약사항

### 9.1 기술 리스크

| 리스크 | 심각도 | 발생 가능성 | 대응 방안 |
|--------|--------|-----------|----------|
| BFS 확산 모델의 물리적 정확도 한계 | 높음 | 높음 | "근사 모델" 고지 명확화. CFD 대비 오차 범위 검증 후 문서화. 과학적 정확도 주장 금지 |
| pgRouting 도로망 불완전성으로 경로 오류 | 중간 | 중간 | OSM 대신 국토부 국가도로망 데이터(MOLIT) 병행 검토. 결과에 "참고용" 고지 |
| DBSCAN 군집 분석 서버 부하 | 낮음 | 중간 | 배치 처리 + 결과 캐싱. 실시간 계산 지양 |
| PostGIS ST_Distance 서울 전역 연산 비용 | 중간 | 높음 | 인접성 사전 계산 테이블 생성 (building_adjacency 테이블). 실시간 쿼리 금지 |

### 9.2 데이터 리스크

| 리스크 | 심각도 | 발생 가능성 | 대응 방안 |
|--------|--------|-----------|----------|
| 위험물 취급소 데이터 오류로 인한 잘못된 위험도 | 높음 | 중간 | 지오코딩 후 수동 샘플 검증. 오류 신고 기능 제공 |
| 건축물대장 structure_type 누락률 (노후 건물 15~25%) | 중간 | 높음 | 누락 건물 명시적 표시. 기본값 적용 시 신뢰도 등급 C 표기 |
| 소방서 위치 데이터 갱신 지연 | 낮음 | 낮음 | 분기 1회 자동 갱신 + 수동 업데이트 기능 |

### 9.3 법적·규제 리스크

| 리스크 | 내용 | 대응 방안 |
|--------|------|----------|
| 소방 위험 정보의 악용 가능성 | 위험 건물 위치 정보가 방화 범죄에 악용될 우려 | 원시 점수 데이터는 인증된 기관만 접근. 일반 공개는 히트맵 수준으로 제한 |
| 시뮬레이션 결과의 법적 책임 | 대피 경로 오류로 피해 발생 시 책임 문제 | 서비스 약관에 "의사결정 지원 도구, 법적 효력 없음" 명시. 현장 판단 우선 고지 |
| 공공데이터 재배포 제약 | 소방청 데이터의 상업적 재배포 라이선스 확인 필요 | 각 데이터 이용 약관 검토 후 B2B API 제공 범위 결정 |

### 9.4 비즈니스 리스크

| 리스크 | 심각도 | 대응 방안 |
|--------|--------|----------|
| 소방청 자체 시스템 구축으로 시장 선점 차단 | 높음 | B2G 진입을 Phase F1 직후로 앞당김. 파일럿 레퍼런스를 통한 조기 관계 구축 |
| 보험사 언더라이팅 시스템 연동 복잡도 | 중간 | REST API + Webhook 표준 인터페이스 우선. 보험사 레거시 시스템 직접 연동은 Phase F4 이후 |
| F0 위험도 모델의 낮은 변별력 | 중간 | 서울 결과: HIGH 40K / MEDIUM 730K / LOW 90건 → MEDIUM 과집중. F1 밀도 보정으로 HIGH 비율 재분류 필요 |

## 10. 구현 로드맵

### Phase별 일정 및 의존성

```
[완료] Phase F0  — 화재 위험도 기본 점수 + 히트맵 (이미 구현)
          ↓
[ 진행 ] Phase F1  — 밀도 위험 보정 + 군집 탐지 + 소방서 데이터
          예상 기간: 4~6주
          선행 조건: 없음 (F0 완료)
          ↓
         Phase F2  — 화재 확산 시뮬레이션 (BFS + 바람 보정)
          예상 기간: 6~8주
          선행 조건: F1 인접 그래프 완료
          ↓
         Phase F3  — 대피 경로 계산 (pgRouting + OSM)
          예상 기간: 4~6주
          선행 조건: F1 완료 (OSM 도로망은 F2와 병행 가능)
          ↓
         Phase F4  — 실시간 기상 & IoT 연동
          예상 기간: 4~6주
          선행 조건: F2, F3 완료
```

### 이정표 (Milestone)

| 이정표 | 내용 | 목표 시점 |
|--------|------|----------|
| M1 | Phase F1 완료 + 자치구 파일럿 제안 | F0 완료 후 6주 |
| M2 | Phase F2 완료 + 보험사 API 데모 | M1 후 8주 |
| M3 | Phase F3 완료 + 첫 번째 B2G 계약 체결 | M2 후 6주 |
| M4 | Phase F4 완료 + SaaS 공개 베타 | M3 후 6주 |
